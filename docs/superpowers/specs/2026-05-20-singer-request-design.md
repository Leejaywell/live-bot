# 主播唱点歌功能设计

## 背景

现有音乐互动功能面向“系统播放点歌”：观众通过礼物或 SC 获得点歌权益，发送 `点歌 歌名` 搜索歌曲，确认后进入系统播放队列。新功能面向“主播演唱请求”：观众提交希望主播唱的歌，主播在后台审核、接单和管理待唱队列。

这两个功能共享同一个“音乐互动”插件入口，但业务账本、申请队列和消耗规则必须独立。现有系统播放点歌逻辑不应因为主播唱功能而改变。

## 目标

- 在音乐互动页面增加“主播唱”Tab。
- 支持观众发送 `主播唱 歌名` 创建主播唱申请。
- 支持独立的主播唱积分和主播唱权益券。
- 礼物/SC 可累计主播唱积分；指定礼物可额外生成权益券。
- 申请创建时不自动扣资源，主播后台接单时再决定扣券、扣积分、改价格或免单。
- 无资源用户也允许提交申请，但标记为无资格。
- OBS 提供小尺寸滚动面板，只展示已接单待唱，不展示待审核申请。

## 非目标

- 不复用现有系统播放点歌队列。
- 不搜索音乐平台候选，不解析播放链接。
- 不触发系统播放器播放。
- 不把主播唱积分和现有点歌积分合并。
- 第一版不做复杂歌单分类、难度定价或自动演唱时长管理。

## 页面入口

音乐互动页面增加 Tab：

- `播放点歌`：保留现有系统播放点歌功能。
- `主播唱`：新增主播唱申请、配置、账本和 OBS 面板管理。

页面入口可以共用现有插件路由，但前端状态、接口调用和展示组件应按 Tab 分开，避免现有页面继续膨胀成单一大组件。

## 指令

新增观众指令：

| 指令 | 示例 | 作用 |
|---|---|---|
| `主播唱 <歌名>` | `主播唱 晴天` | 创建主播唱申请 |
| `我的主播唱` | `我的主播唱` | 查询自己的主播唱申请状态 |
| `我的唱歌积分` | `我的唱歌积分` | 查询主播唱积分和权益券 |

`主播唱 <歌名>` 只保存文本歌名，不搜索音乐平台候选。歌名去除首尾空白后必须非空。后续如果需要备注，可扩展为 `主播唱 歌名 @备注` 或后台编辑，不进入第一版。

## 资源模型

主播唱使用两种独立资源。

### 主播唱积分

- 礼物和 SC 按金额累计，默认 `1 金额 = 1 积分`。
- 积分不过期。
- 指定礼物命中权益券规则时，仍然照常累计积分。
- 积分只用于主播唱接单，不影响现有点歌积分。

### 主播唱权益券

- 由指定礼物规则额外生成。
- 权益券不过期。
- 一张权益券可抵扣一次主播唱接单。
- 接单时主播可以选择不用券，改为扣积分或免单。

### 指定礼物规则

配置项至少包含：

- 礼物名称。
- 最小数量。
- 生成权益券数量。
- 是否启用。

命中规则时，系统同时执行：

1. 按礼物金额累计主播唱积分。
2. 按规则生成主播唱权益券。

如果多条规则都匹配同一礼物，第一版采用“按配置顺序命中第一条启用规则”的策略，避免重复发券。

## 申请状态

主播唱申请状态：

```text
pending   待审核
accepted  已接单待唱
singing   正在唱
done      已唱
rejected  已拒绝
deleted   已删除
```

状态流：

```text
pending -> accepted -> singing -> done
pending -> rejected
pending -> deleted
accepted -> deleted
singing -> deleted
```

`done` 状态不允许删除，只进入历史。需要纠错时可以在后台增加备注，但第一版不做撤销。

## 申请创建规则

观众发送 `主播唱 歌名` 后：

1. 创建一条 `pending` 申请。
2. 记录用户 UID、昵称、歌名、房间、当前直播场次、提交时间。
3. 记录资源快照：当前可用主播唱积分、可用权益券数量、默认接单价格。
4. 根据快照标记资格：
   - 有至少 1 张权益券，标记为有资格。
   - 或积分大于等于默认接单价格，标记为有资格。
   - 否则标记为无资格。

无资格申请仍进入待审核列表，但不进入 OBS 面板。主播可以拒绝、删除或免单接单。

## 接单与扣减

资源扣减只发生在主播后台接单时。

后台接单动作：

- `用权益券接单`：扣 1 张权益券，申请进入 `accepted`。
- `按积分接单`：扣默认价格积分，申请进入 `accepted`。
- `改价格接单`：主播输入本次价格，扣对应积分，申请进入 `accepted`。
- `免单接单`：不扣资源，申请进入 `accepted`。
- `拒绝`：申请进入 `rejected`，不扣资源。
- `删除`：申请进入 `deleted`，不扣资源。

如果接单时资源已经不足，例如观众在另一个请求里先消耗了券或积分，系统应阻止扣减并提示主播重新选择接单方式。免单接单不受资源限制。

已接单后不自动退款。删除 `accepted` 或 `singing` 请求也不退款，第一版用操作提示说明这一点；后续可以增加手动退款功能。

## 后台功能

`主播唱` Tab 分为四块。

### 配置区

- 功能开关。
- 默认接单积分价格。
- 礼物/SC 积分倍率。
- 指定礼物权益券规则。
- OBS 小面板 URL。

### 待审核

展示：

- 用户昵称和 UID。
- 歌名。
- 提交时间。
- 当前快照积分、快照权益券。
- 资格状态。

操作：

- 用权益券接单。
- 按积分接单。
- 改价格接单。
- 免单接单。
- 拒绝。
- 删除。

### 已接单待唱

展示：

- 队列序号。
- 用户昵称。
- 歌名。
- 消耗方式。
- 接单时间。

操作：

- 置顶。
- 调整排序。
- 标记正在唱。
- 标记已唱。
- 删除。

### 历史

展示已唱、已拒绝、已删除记录，用于查账。历史列表默认按更新时间倒序。

## OBS 小面板

新增 OBS 入口，例如：

```text
GET /song-request/singer-panel
GET /song-request/api/singer-panel
```

展示规则：

- 只展示 `accepted` 请求。
- 不展示 `pending`、`rejected`、`deleted`、`done`。
- `singing` 属于后台演唱状态，不进入待唱滚动列表。
- 面板保持小尺寸滚动展示。
- 单项内容只显示队列序号、昵称、歌名。

OBS 面板不提供审核操作，只读后台已接单队列。

## 数据设计

建议新增独立表，而不是复用 `song_requests` 和 `song_request_credits`。

### singer_request_points

记录主播唱积分流水。

关键字段：

- `id`
- `session_id`
- `room_id`
- `uid`
- `uname`
- `delta`
- `source_type`
- `source_event_id`
- `created_at`

当前积分余额通过流水求和，或后续增加聚合表优化。第一版可以用流水求和，便于查账。

### singer_request_coupons

记录主播唱权益券。

关键字段：

- `id`
- `session_id`
- `room_id`
- `uid`
- `uname`
- `source_type`
- `source_event_id`
- `used_request_id`
- `created_at`
- `used_at`

权益券不过期。可用券是 `used_at is null` 的数量。

### singer_requests

记录主播唱申请。

关键字段：

- `id`
- `session_id`
- `room_id`
- `uid`
- `uname`
- `song_title`
- `status`
- `qualified_snapshot`
- `points_snapshot`
- `coupons_snapshot`
- `default_price_snapshot`
- `accepted_price`
- `consume_type`
- `consume_ref_id`
- `sort_order`
- `created_at`
- `accepted_at`
- `updated_at`

`consume_type` 可取：

- `coupon`
- `points`
- `free`
- `none`

`consume_ref_id` 用于关联被消耗的券或积分流水。

## 后端组件

在 `src/music/` 下新增主播唱子模块，命名建议为 `singer_request`。

职责拆分：

- `command.rs`：解析 `主播唱`、`我的主播唱`、`我的唱歌积分`。
- `storage.rs`：主播唱表结构、积分/券/申请的 SQL 操作。
- `service.rs`：处理直播事件、观众指令、后台动作。
- `types.rs`：后台和前端 API 结构。

现有 `MusicInteractionService` 可以在直播事件入口中同时调用系统点歌服务和主播唱服务，但两者内部不共享账本。

## 设置项

在 `MusicInteractionSettings` 中增加主播唱配置分组：

- `SingerRequestEnabled`
- `SingerRequestDefaultPrice`
- `SingerRequestPointMultiplier`
- `SingerRequestCouponGiftRules`
- `SingerRequestPanelEnabled`
- `SingerRequestPanelMaxItems`

默认值：

- 功能默认关闭。
- 默认接单价格 `100`。
- 积分倍率 `1.0`。
- 指定礼物规则为空。
- OBS 面板默认开启，最多展示 `8` 条。

## 错误处理

- 存储未就绪：观众指令返回“主播唱功能暂未就绪”。
- 功能未开启：返回简短提示“主播唱点歌暂未开启”。
- 歌名为空：提示“请发送 主播唱 歌名”。
- 接单资源不足：后台提示资源不足，不改变申请状态。
- 重复操作：如果申请状态已变化，后台提示刷新列表。
- OBS 接口读取失败：返回空列表，页面显示等待数据。

## 测试计划

单元测试：

- 解析 `主播唱 晴天`。
- 礼物/SC 累计主播唱积分，不影响现有点歌积分。
- 指定礼物同时累计积分和生成权益券。
- 无资源用户提交申请后状态为 `pending` 且资格为无资格。
- 有券用户接单时消耗 1 张券。
- 有积分用户改价格接单时扣对应积分。
- 资源不足时接单失败且申请仍为 `pending`。
- OBS API 只返回 `accepted`，不返回 `pending`、`singing`、`rejected`、`deleted`、`done`。

集成检查：

- `cargo test singer_request`
- `npm run build`
- 前端页面烟测：音乐互动页能切到“主播唱”Tab，能看到待审核、待唱和配置区。

## 已确认的产品决策

- 使用方案 2：挂在现有音乐互动入口，用 Tab 区分。
- 主播唱是完全独立队列。
- 同时支持金额积分和指定礼物权益券。
- 指定礼物命中时仍然累计积分。
- 观众申请只收文本歌名，不搜索音乐平台。
- 主播后台接单时才扣资源。
- 无资格申请允许进入待审核。
- OBS 只展示已接单待唱，小面板滚动展示。
- 后台操作包含接单、免单接单、改价格、拒绝、标记已唱、置顶排序和删除。
