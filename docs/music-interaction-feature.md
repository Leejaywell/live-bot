# 音乐互动点歌功能整理

本文档整理当前「音乐互动」点歌功能的产品规则、配置项、用户流程、队列机制、OBS 页面和后续增强方向。

## 1. 功能定位

音乐互动用于把直播间礼物、醒目留言和弹幕点歌串成一条闭环：

1. 观众通过送礼或醒目留言获得点歌权益。
2. 观众发送点歌指令，系统搜索歌曲并返回候选。
3. 观众确认候选后，系统消耗一条点歌权益并写入队列。
4. 主播在插件中心查看队列，也可以打开系统播放器 URL scheme 播放歌曲。
5. OBS 或其他直播软件捕捉点歌单网页，把当前播放、队列和排行展示到直播画面。

设计目标是让点歌足够方便、确认足够精准、礼物档位可控，并给主播保留调价、装饰直播间和控制直播节奏的空间。

## 2. 已实现入口

### 插件中心

插件中心里有「音乐互动」菜单，用于：

- 开关音乐互动功能。
- 获取 OBS 点歌单 URL。
- 搜索歌曲并模拟确认。
- 查看当前队列。
- 配置 OBS 展示样式。
- 配置礼物金额档位。
- 配置点歌播放方式。

### OBS 页面

当前 OBS 主入口是一个统一点歌页面，页面内部可以同时显示多个面板，主播在设置界面控制每个面板开启或关闭。

```text
GET /song-request
GET /song-request/playlist
```

兼容入口仍然保留：

```text
GET /song-request/now-playing
GET /song-request/rank
```

配套 JSON 接口：

```text
GET /song-request/api/queue
GET /song-request/api/now-playing
GET /song-request/api/rank
```

插件中心生成的默认 URL 是统一点歌页面：

```text
http://127.0.0.1:<port>/song-request/playlist
```

直播软件可以把这个地址作为浏览器源捕捉。

## 3. 观众点歌流程

### 3.1 送礼获得权益

观众送礼或发送醒目留言后，系统按礼物金额匹配一个点歌档位，写入 `song_request_credits`。

当前规则：

- 礼物金额低于所有启用档位门槛时，不产生点歌权益。
- 达到多个档位时，取门槛最高的启用档位。
- 权益与当前直播场次、房间、用户绑定。
- 权益未被使用且未过期时，才可用于确认点歌。

### 3.2 搜索歌曲

观众发送：

```text
点歌 晴天 周杰伦
```

系统会搜索歌曲并返回候选列表。候选上下文保存到 `song_search_contexts`，用于后续 `#编号` 确认。
候选列表必须展示播放源，例如 `[网易云] 晴天 - 周杰伦`，避免用户确认后才发现来源不符合主播设置。

搜索源受主播「播放方式」限制：

- `按歌曲来源`：允许所有已接入搜索源。
- `网易云音乐`：只搜索网易云，确认时也只允许网易云候选入队。
- `QQ 音乐`：只搜索 QQ 音乐，确认时也只允许 QQ 音乐候选入队。
- `浏览器`：不限制音乐平台来源，但播放时走网页链接。

当前已接入搜索 Provider 是网易云；如果主播选择 QQ 音乐，系统会提示该平台搜索源尚未接入，而不是返回网易云候选。

当前支持的指令：

| 指令 | 示例 | 作用 |
|---|---|---|
| `点歌 <关键词>` | `点歌 晴天 周杰伦` | 搜索歌曲，返回候选 |
| `点歌 #<编号>` | `点歌 #1` | 确认最近一次搜索结果 |
| `点歌 #<编号> 确认` | `点歌 #2 确认` | 确认最近一次搜索结果 |
| `确认 <编号>` | `确认 1` | 确认最近一次搜索结果 |
| `选 <编号>` | `选 2` | 确认最近一次搜索结果 |
| `换一批` | `换一批` | 预留为换候选 |
| `我的点歌` | `我的点歌` | 查询自己的点歌状态 |
| `我的积分` | `我的积分` | 查询可用点歌积分、当前档位和惩罚提醒 |
| `点歌积分` | `点歌积分` | 查询可用点歌积分、当前档位和惩罚提醒 |
| `我的档位` | `我的档位` | 查询可用点歌积分、当前档位和惩罚提醒 |
| `点歌档位` | `点歌档位` | 查询可用点歌积分、当前档位和惩罚提醒 |
| `取消点歌` | `取消点歌` | 取消自己的未播放点歌 |

### 3.3 用户确认歌曲

确认时系统会校验：

- 当前有直播场次。
- 用户 UID 和昵称有效。
- 用户最近有未过期的搜索候选。
- 候选编号存在。
- 用户有未使用、未过期的点歌权益。

确认成功后：

1. 取该用户当前可用权益。
2. 按权益档位计算队列优先级。
3. 写入 `song_requests`。
4. 把对应 `song_request_credits` 标记为已使用。
5. 返回 `已加入点歌队列 #<request_id>`。

如果没有权益，系统返回：

```text
还没有可用点歌权益，请先送礼解锁
```

### 3.4 查询积分、档位和惩罚提醒

观众可以发送 `我的积分`、`点歌积分`、`我的档位` 或 `点歌档位` 查询自己的点歌权益。

系统会返回：

- 当前可用点歌积分。
- 当前已解锁的最高启用档位。
- 距离下一档还差多少。
- 连续重复点歌会降低排序的提醒。
- 确认点歌会消耗可用权益的提醒。

示例：

```text
你当前可用点歌积分 100，当前档位：优先点歌，再送 133 可到 插队。提醒：连续重复点歌会降低排序，确认点歌会优先消耗可用权益。
```

这里的“积分”来自未使用、未过期的点歌权益。当前确认点歌仍按可用权益消耗，不会无限保留。

## 4. 礼物金额档位

档位现在是可配置的。固定 5 个业务档位 ID，配置项可调整：

- `Name`：展示名称。
- `MinCredit`：最低礼物金额。
- `BaseScore`：队列基础分。
- `Enabled`：是否启用该档位。

默认配置：

| 档位 ID | 默认名称 | 最低金额 | 基础分 |
|---|---|---:|---:|
| `ordinary` | 普通点歌 | 10 | 1000 |
| `priority` | 优先点歌 | 66 | 3000 |
| `jump_queue` | 插队 | 233 | 6000 |
| `exclusive` | 专属点歌 | 520 | 9000 |
| `playlist_takeover` | 包场歌单 | 1999 | 12000 |

### 4.1 档位匹配规则

假设当前配置如下：

```text
普通点歌 >= 10
优先点歌 >= 66
插队 >= 233
专属点歌 >= 520
包场歌单 >= 1999
```

则：

| 礼物金额 | 命中档位 |
|---:|---|
| 9 | 无权益 |
| 10 | 普通点歌 |
| 66 | 优先点歌 |
| 300 | 插队 |
| 520 | 专属点歌 |
| 2000 | 包场歌单 |

如果某个档位被禁用，该档位不会被命中，系统会继续匹配其他启用档位。

### 4.2 基础分含义

基础分不是礼物金额，而是队列排序权重。当前排序公式是：

```text
最终排序分 = 档位基础分 + 礼物金额封顶值 + 粉丝加成 - 重复点歌惩罚
```

当前已接入的是：

```text
最终排序分 = BaseScore + min(礼物金额, 2000)
```

`fan_bonus` 和 `repeat_count` 参数已预留，但当前确认点歌路径传入的是 `0`。产品文案上已经提醒“连续重复点歌会降低排序”；后续接入真实连续点歌计数后，`repeat_count` 会参与扣分。

示例：

| 用户 | 档位 | 礼物金额 | 基础分 | 最终排序分 |
|---|---|---:|---:|---:|
| A | 普通点歌 | 100 | 1000 | 1100 |
| B | 优先点歌 | 66 | 3000 | 3066 |
| C | 插队 | 233 | 6000 | 6233 |

因此高档位会稳定排在低档位前面。基础分间隔越大，档位越强；基础分间隔越小，大额低档位越有机会追上高档位。

## 6. 队列排序机制

队列数据存储在 `song_requests`。

当前队列展示排序：

```text
播放中歌曲优先
然后按 priority_score 从高到低
同分按 request_id 从小到大
```

也就是说：

1. 正在播放的歌永远在最上面。
2. 高档位、高基础分、高礼物金额的请求更靠前。
3. 同分时先确认的人更靠前。

## 7. 插队、专属点歌、包场歌单

当前实现里，这三类已经作为礼物档位和视觉标识存在：

| 机制 | 当前已实现 | 后续可增强 |
|---|---|---|
| 插队 | 队列硬排序优先于普通/优先点歌，OBS 和控制台显示插队档位 | 强制插到当前歌曲后第 1 位或第 2 位 |
| 专属点歌 | 队列硬排序优先于插队，OBS 可显示专属档位名称和高亮 | 专属感谢动画、TTS 口播、指定平台/版本 |
| 包场歌单 | 队列最高优先级，OBS 显示包场标识和冠名文案 | 连续多首、时段包场、主播确认后执行 |

为了避免直播节奏失控，建议后续把“插队位置”和“包场连续歌曲数”做成单独配置，而不是只靠基础分。

当前队列排序有两层保障：

1. 档位硬优先级：包场歌单 > 专属点歌 > 插队 > 优先点歌 > 普通点歌。
2. 同档位内部再按 `priority_score` 和确认顺序排序。

这样即使主播把某个低档位基础分配置得很高，高价权益仍然不会被低档位反超。

## 8. 播放控制

主播在队列中打开某首歌时，系统会根据歌曲来源生成系统 URL scheme：

| 来源 | 打开方式 |
|---|---|
| 网易云 | `orpheus://song/{song_id}` |
| QQ 音乐 | `qqmusic://song/{song_id}` |
| 其他 | `https://music.163.com/#/song?id={song_id}` |

打开成功后，该点歌请求会标记为 `playing`。

主播也可以在「音乐互动」设置里选择播放方式：

| 播放方式 | 行为 |
|---|---|
| 按歌曲来源 | 搜索所有已接入平台，播放时网易云歌曲用网易云 URL scheme，QQ 音乐歌曲用 QQ 音乐 URL scheme |
| 网易云音乐 | 只搜索网易云，确认和播放都限制为网易云，使用 `orpheus://song/{song_id}` |
| QQ 音乐 | 只搜索 QQ 音乐，确认和播放都限制为 QQ 音乐，使用 `qqmusic://song/{song_id}` |
| 浏览器 | 不限制搜索来源，播放时使用网页链接打开 |

默认是“按歌曲来源”，尽量匹配歌曲本身的平台。强制选择某个播放器时，需要主播本机已安装对应客户端，并且该播放器能识别对应歌曲 ID。
如果主播修改播放方式，旧候选在确认时会再次校验来源；旧队列歌曲在打开时也会校验来源，避免用错误播放器打开不兼容的歌曲 ID。

### 8.1 播放状态和手动控制

当前第一期播放控制以“主播可接管”为核心，队列支持以下状态：

| 状态 | 含义 |
|---|---|
| `queued` | 排队中 |
| `playing` | 播放中 |
| `finished` | 主播标记已播完 |
| `skipped` | 主播跳过 |
| `failed` | 主播标记失败 |

插件中心队列提供：

- 播放：打开播放器并标记为 `playing`。
- 播完：标记为 `finished`。
- 跳过：标记为 `skipped`。
- 失败：标记为 `failed`。

当主播把一首歌标记为 `finished` 时，如果「播放模式」设置为“播完后自动下一首”，系统会自动查找下一首 `queued` 歌曲并打开。

### 8.2 播放模式

主播可以在「音乐互动」设置里选择：

| 播放模式 | 行为 |
|---|---|
| 必须手动确认 | 主播需要手动点击播放，也需要手动标记播完、跳过或失败 |
| 播完后自动下一首 | 主播标记当前歌曲播完后，系统自动打开下一首排队歌曲 |

当前实现没有假装 URL scheme 能知道播放器真实结束。原因是系统 URL scheme 只能证明系统调用成功，不能证明播放器已经开始播放或播放完毕。

后续要做“当前播放结束检测”，建议按以下顺序实现：

1. 用 `duration_ms` 估算 `expected_end_at`，到点后提示或自动完成。
2. 针对网易云、QQ 音乐做播放器适配器，检测进程、系统媒体状态和曲目信息。
3. 对能拿到播放 URL 的平台提供内置 Web 播放器，使用 `ended` 事件做强检测。

### 8.3 打开成功校验

当前已做：

- `open_url` 返回错误时，不标记播放成功。
- 来源不匹配当前播放方式时，拒绝打开。
- 打开成功后标记为 `playing`。

尚未做：

- 检测播放器进程是否出现。
- 检测系统媒体状态是否为播放中。
- 检测正在播放曲目是否匹配点歌名称和歌手。

这些需要按播放器分别实现 adapter，不能只靠 URL scheme。

## 9. OBS 皮肤和展示配置

当前音乐互动配置包含：

- `Skin`：皮肤。
- `Player`：播放方式。
- `Transparent`：透明背景。
- `Width` / `Height`：显示尺寸。
- `ShowCover`：是否显示封面/唱片视觉。
- `ShowRequester`：是否显示点歌人。
- `ShowGiftTier`：是否显示礼物档位。
- `ShowQueue`：是否显示队列。
- `ShowNowPlayingPanel`：统一点歌页是否显示当前播放面板。
- `ShowQueuePanel`：统一点歌页是否显示队列面板。
- `ShowRankPanel`：统一点歌页是否显示排行面板。
- `ShowTodayValue`：是否显示金额。
- `PrimaryColor`：主色。
- `FontScale`：字号缩放。

当前已有的点歌单视图：

| 页面 | 用途 |
|---|---|
| `/song-request` | 统一点歌页，按设置显示多个面板 |
| `/song-request/playlist` | 统一点歌页，按设置显示多个面板 |
| `/song-request/now-playing` | 兼容入口，只展示当前播放 |
| `/song-request/rank` | 兼容入口，只展示点歌排行 |

当前已有的皮肤方向：

| 皮肤 | 风格 |
|---|---|
| `neon` | 霓虹高能，适合 PK 和高价插队 |
| `idol-stage` | 舞台应援，适合虚拟主播、唱歌直播 |
| `vinyl` | 黑胶唱片感，适合音乐陪伴、安静直播间 |

## 10. 金额统计口径

设计上需要支持两个口径：

| 口径 | 作用 |
|---|---|
| 本场 | 控制当前队列、展示本场收益、刺激实时消费 |
| 当天 | 做日榜、日报、活动复盘 |

当前队列和 OBS 主要围绕当前直播场次读取数据。后续如果要做当天统计，应继续补齐自然日聚合、日榜切换和后台统计面板。
